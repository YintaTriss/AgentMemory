## Bug 1 — TypeError on None input to compress()

**Lines:** 34  
**Severity:** P0

**Description:**  
`if not memories:` correctly handles the empty-list case, returning early. However, if `memories` is explicitly `None` (not a list), the guard `if not memories` raises `TypeError: object of type 'NoneType' has no len()`. The function's docstring does not specify that `None` is handled; callers passing `None` will get an unhandled crash.

**Fix:**  
Change line 34 to `if not memories:` and add a `None` guard before it:

```python
if memories is None:
    return "No relevant memories found."
if not memories:
    return "No relevant memories found."
```

Or consolidate:
```python
if not memories:
    return "No relevant memories found."
memories = memories or []
```

## Bug 2 — AttributeError if memory item is not a dict

**Lines:** 36–46  
**Severity:** P0

**Description:**  
The loop `for mem in memories:` calls `mem.get("meta", {}).get("importance", 0.5)` on line 45. If any item in `memories` is not a `dict` (e.g., a string, `None`, or a custom object), `mem.get(...)` will raise `AttributeError`. There is no type validation. This is especially dangerous because the bare `except:` on line 53–54 silently swallows this error and returns `datetime.min`, masking the corruption.

**Fix:**  
Add type validation inside the loop:

```python
for mem in memories:
    if not isinstance(mem, dict):
        continue  # skip invalid items
    importance = mem.get("meta", {}).get("importance", 0.5)
    ...
```

Or raise an explicit error:
```python
for mem in memories:
    if not isinstance(mem, dict):
        raise TypeError(f"Memory item must be a dict, got {type(mem).__name__}")
```

## Bug 3 — Bare `except:` catches all exceptions

**Lines:** 53–54  
**Severity:** P1

**Description:**  
```python
try:
    return datetime.fromisoformat(ts)
except:
    return datetime.min
```
The bare `except:` clause catches `ValueError` (the expected `datetime.fromisoformat` failure), but also catches `AttributeError`, `TypeError`, `KeyboardInterrupt`, `SystemExit`, and any other exception. This masks real bugs. In the context of Bug 2, this causes invalid memory items to silently sort to the end instead of surfacing an error.

**Fix:**  
Catch only the specific exception:

```python
try:
    return datetime.fromisoformat(ts)
except ValueError:
    return datetime.min
```

## Bug 4 — English punctuation not handled in `extract_facts()`

**Lines:** 76  
**Severity:** P1

**Description:**  
```python
sentences = content.replace("！", "。").replace("？", "。").split("。")
```
Only Chinese full-width punctuation (`！`, `？`) is normalized to `。` before splitting. English punctuation (`!`, `?`) and other sentence-ending characters are not handled. If the input `content` is in English or uses mixed punctuation, `extract_facts()` will fail to split sentences properly, leading to either zero facts extracted or very long concatenated strings.

**Fix:**  
Expand the replacement to include English punctuation:

```python
sentences = content.replace("!", "。").replace("？", "。").replace("！", "。").replace("?", "。").split("。")
```

Or use a regex-based approach:
```python
import re
sentences = re.split(r'[。.!?！？]', content)
```

## Bug 5 — Keyword list in `extract_facts()` is Chinese-only

**Lines:** 79–80  
**Severity:** P1

**Description:**  
```python
if any(kw in sent for kw in ["应该", "必须", "重要", "决定", "计划", "完成", "成功", "失败"]):
    facts.append(sent)
```
All keywords are Chinese. If the input `content` is in English (or any non-Chinese language), `extract_facts()` will always return an empty list regardless of content importance. This makes the method effectively broken for English inputs.

**Fix:**  
Add English keywords, or make the detection language-agnostic:

```python
chinese_kw = ["应该", "必须", "重要", "决定", "计划", "完成", "成功", "失败"]
english_kw = ["should", "must", "important", "decided", "plan", "completed", "success", "failed", "need to", "remember"]
if any(kw in sent for kw in chinese_kw + english_kw):
    facts.append(sent)
```

## Bug 7 — No validation of memory dict schema (id, content, meta)

**Lines:** 35–46  
**Severity:** P2

**Description:**  
The docstring promises that memory dicts contain `'id'`, `'content'`, and `'meta'` keys, but the code never validates this. Missing keys are handled silently with `.get(key, default)` — for example, `mem.get("content", "")` silently defaults to empty string. This means corrupted or malformed memories produce silently corrupted output rather than raising errors.

**Fix:**  
Add a schema validation helper and validate each memory:

```python
REQUIRED_KEYS = {"id", "content", "meta"}

def _validate_memory(mem: Dict) -> None:
    if not isinstance(mem, dict):
        raise TypeError("Memory must be a dict")
    missing = REQUIRED_KEYS - set(mem.keys())
    if missing:
        raise ValueError(f"Memory missing required keys: {missing}")

# In compress():
for mem in memories:
    _validate_memory(mem)
    ...
```

## Bug 8 — Truncation cuts mid-sentence

**Lines:** 71–72  
**Severity:** P2

**Description:**  
```python
if len(result) > self.max_context_chars:
    result = result[:self.max_context_chars - 100] + "\n\n... (truncated)"
```
When truncation occurs, it cuts at an arbitrary character boundary, potentially slicing through the middle of a word, tag, or line. The `max_context_chars` is supposed to be a quality gate but the truncation logic doesn't respect sentence or line boundaries.

**Fix:**  
Truncate at a line boundary instead:

```python
if len(result) > self.max_context_chars:
    truncated = result[:self.max_context_chars - 100]
    # Find the last newline or sentence end
    last_newline = truncated.rfind("\n")
    if last_newline > self.max_context_chars * 0.7:
        truncated = truncated[:last_newline]
    result = truncated + "\n\n... (truncated)"
```
