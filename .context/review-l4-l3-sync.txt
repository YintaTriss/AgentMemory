## BUGS FOUND

### BUG-1
- Line: sync.py 153
- Severity: P0
- Title: delete_from_l3() always returns True, silently swallowing failures
- Description:
    The method wraps l3.delete() in a try/except that catches ALL exceptions,
    then unconditionally returns True. Even when l3.delete() returns False
    (meaning the record was not found in LanceDB), delete_from_l3() still
    returns True. This masks deletion failures.
- Code:
    def delete_from_l3(self, memory_id: str) -> bool:
        try:
            self.l3.delete(memory_id)   # ← return value silently discarded
            vec_path = ...
            if vec_path.exists():
                vec_path.unlink()
            return True                 # ← always True, even on failure
        except Exception as e:
            print(f"[Sync] Error deleting {memory_id} from L3: {e}")
            return False                # ← only False on exception
- Fix:
    Capture and propagate the actual return value:
    def delete_from_l3(self, memory_id: str) -> bool:
        try:
            vec_path = self.memory_dir / f"{memory_id}.vec.json"
            if vec_path.exists():
                vec_path.unlink()
            return self.l3.delete(memory_id)  # propagate the bool
        except Exception as e:
            print(f"[Sync] Error deleting {memory_id} from L3: {e}")
            return False

### BUG-2
- Line: sync.py 151
- Severity: P1
- Title: _write_vec_json called BEFORE l3.upsert — vec.json written even if L3 insert fails
- Description:
    In sync_one(), the sequence is:
      1. l3.upsert(...)   ← can fail/return False silently
      2. _write_vec_json(...)  ← always executes even if upsert failed
    This leaves a orphaned .vec.json file when L3 upsert fails.
    Bug-1 amplifies this: delete_from_l3() won't clean up the orphaned vec
    because it returns True regardless.
- Fix:
    Verify upsert succeeded before writing vec.json:
    success = self.l3.upsert(...)
    if success:
        self._write_vec_json(memory_id, vector)
    NOTE: l3.upsert() currently has no return value. It should be modified to
    return bool on success/failure, then the above guard applied.

### BUG-3
- Line: manager.py 64
- Severity: P2
- Title: manager.add() has no content validation
- Description:
    add() generates a memory ID and calls l4.save() even when content is
    empty string "". This creates a phantom memory with no content, wasting
    storage and polluting the vector DB.
- Fix:
    Add early validation in add():
    if not content or not content.strip():
        raise ValueError("content cannot be empty")

### BUG-4
- Line: sync.py 45
- Severity: P1
- Title: No null check on embedder output before passing to l3.upsert
- Description:
    vector = self._embed_fn(content)  # could theoretically return None
    self.l3.upsert(..., vector=vector, ...)
    If embed() returns None (e.g., empty string edge case), this passes None
    to LanceDB which expects List[float]. This could corrupt the vector DB
    or cause a runtime error.
- Fix:
    After embedding:
    if not vector or not isinstance(vector, list):
        print(f"[Sync] Embedding failed for {memory_id}")
        return False

### BUG-5
- Line: l3_lancedb.py 169-171
- Severity: P1
- Title: LanceDB delete path swallows exception and STILL returns True
- Description:
    try:
        self._table.delete(f"id = '{id}'")
        return True
    except Exception as e:
        print(f"[L3] Delete error: {e}")
        return False
    If delete() raises ANY exception (table locked, schema mismatch, etc.),
    it prints and returns False. However, there is a comment "P0-4 fix: main
    path must return bool, not None" suggesting the original bug was that
    the LanceDB delete() DID NOT have a return statement. The current code
    DOES return True on success and False on exception, so this appears
    already fixed. Listed here for completeness.
- Fix: Already fixed. No action needed.
